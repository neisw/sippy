# TRT-2588: Config Hierarchy for Explicit Job-to-Release Mapping

**Date:** 2026-04-21
**JIRA:** [TRT-2588](https://redhat.atlassian.net/browse/TRT-2588)
**Author:** Stephen Goeddel

## Problem Statement

Synthetic releases like `rosa-stage` cannot reliably claim jobs that contain OCP
version numbers in their names. A job like
`periodic-ci-openshift-osde2e-main-nightly-4.22-osd-aws` is explicitly listed
under `rosa-stage` in `config/openshift.yaml`, but it gets assigned to the
`4.22` OCP release instead because the matching logic extracts the version
number from the job name.

Users of the rosa-stage dashboard see an incomplete view of their jobs, while
those jobs silently appear under OCP release dashboards where they don't belong.

## Root Cause Analysis

There are **two independent matching systems** that both need to agree on which
release a job belongs to, and both have the same bug:

### 1. Prow Loader: `processProwJob` (pkg/dataloader/prowloader/prow.go:533)

This function determines which release a job run is imported under in the
database. It iterates `pl.releases` (a `[]string`) and returns on the **first
match**, checking:

1. Exact match in `config.Releases[release].Jobs[jobName]`
2. Regexp match against `config.Releases[release].Regexp[]`

Since the job `periodic-ci-openshift-osde2e-main-nightly-4.22-osd-aws` is
listed in both `rosa-stage.Jobs` AND `4.22.Jobs` (added by the config
generator), whichever release is iterated first wins. The iteration order of
`pl.releases` is externally determined and not guaranteed to favor synthetic
releases.

### 2. Variant Extraction: `setRelease` (pkg/variantregistry/ocp.go:677)

This function determines the `Release` variant for a job. Its logic:

```text
1. Check if job is in any release's Jobs map AND release name is in job name
2. Extract version numbers from job name (e.g., "4.22")
3. Set VariantRelease to the extracted version (UNCONDITIONALLY overwrites step 1)
```

Step 1 fails for rosa-stage because `strings.Contains("periodic-ci-...-4.22-osd-aws", "rosa-stage")` is false. Step 3 then extracts `4.22` from the name and sets that as the release.

Even if step 1 succeeded, step 3 would overwrite it unconditionally -- the
extraction at line 692-695 runs regardless and always sets the release to the
last version number found in the job name.

### 3. Variant Snapshot: `Identify` (pkg/variantregistry/snapshot.go:31)

The variant snapshot iterates all releases and all jobs, calling
`calculateVariantsForJob` for each. A job listed in multiple releases' `Jobs`
maps gets processed multiple times with non-deterministic results (Go map
iteration order).

## Proposed Solution

### Two Data Sources, One Override Map

The solution combines data from two independent sources:

1. **BigQuery `Releases` table** — Contains a `Synthetic` boolean column that
   identifies which releases are synthetic (e.g., `rosa-stage`, `aro-integration`).
   This is the source of truth for which releases are synthetic.

2. **YAML config (`config/openshift.yaml`)** — Contains job-to-release mappings
   in each release's `Jobs` map. This file is auto-generated by
   `sippy-config-generator` and must not be manually modified.

At initialization, a standalone function combines these two inputs to produce a
`map[string]string` (jobName → releaseName) of synthetic release job overrides.
This map is then passed as a parameter to all consumers that need it.

### What Is a Synthetic Release?

Synthetic releases are releases that don't correspond to a specific OCP version
— they represent cross-cutting concerns like managed service testing
(`rosa-stage`, `aro-integration`) or automation tooling (`automation`).

Because synthetic releases don't own a version number, their jobs often contain
OCP version strings (e.g., `-4.22-`) that would otherwise cause name-based
matching to steal them into an OCP release. Marking a release as synthetic in
BigQuery tells the matching system that its job associations take priority over
name-based version extraction.

The current synthetic releases are: `aro-integration`, `aro-production`,
`aro-stage`, `automation`, `ocp-hypershift`, `rosa-integration`,
`rosa-production`, `rosa-stage`, and `rrp-integration`.

Note: `Presubmits` is **not** a synthetic release. It uses regexp-based matching
and has no explicit `Jobs` entries that need override protection.

### Pre-Computed Synthetic Release Job Overrides Map

```go
// Standalone function in pkg/apis/config/v1/types.go
func BuildSyntheticReleaseJobOverrides(
    releases map[string]ReleaseConfig,
    syntheticReleaseNames []string,
) (map[string]string, error) {
    syntheticSet := make(map[string]bool, len(syntheticReleaseNames))
    for _, name := range syntheticReleaseNames {
        syntheticSet[name] = true
    }
    overrides := map[string]string{}
    for releaseName, releaseCfg := range releases {
        if !syntheticSet[releaseName] {
            continue
        }
        for jobName := range releaseCfg.Jobs {
            if existing, conflict := overrides[jobName]; conflict {
                return nil, fmt.Errorf(
                    "job %q is claimed by synthetic releases %q and %q",
                    jobName, existing, releaseName,
                )
            }
            overrides[jobName] = releaseName
        }
    }
    return overrides, nil
}
```

A job appearing in both a synthetic and standard release is expected and fine --
the synthetic release wins. The error case is a job claimed by two *different*
synthetic releases, which would be a config mistake with no clear winner.

### Why an Explicit Flag Instead of Auto-Detection

All current synthetic releases have non-version names (rosa-stage,
aro-integration, etc.) while all OCP releases are version numbers, so
auto-detection via regex is tempting. However, an explicit field is preferred:

- **Transparent.** The BigQuery Releases table clearly marks which releases have
  special matching semantics.
- **Future-proof.** If a release name ever breaks the naming convention
  (e.g., `rosa-4.22-stage`), the field still works.
- **Opt-in.** New synthetic releases must deliberately declare themselves,
  preventing accidental priority changes.

### Why Release-Level (Not Per-Job)

- All existing synthetic releases need ALL their jobs pinned, not a subset.
- Keeps `Jobs` as `map[string]bool` -- no structural config change needed.
- The config generator doesn't need to change; it can continue adding these
  jobs to OCP releases. The synthetic flag ensures the synthetic release wins.
- If a future use case needs per-job granularity, we can add it then.

### Why BigQuery (Not YAML Config)

The YAML config file (`config/openshift.yaml`) is entirely auto-generated by
`sippy-config-generator` in `openshift/ci-tools`. It cannot be manually modified
— any changes would be overwritten on the next generation cycle.

The BigQuery `Releases` table already stores release metadata (versions, GA
dates, capabilities) that is maintained independently of the auto-generated
config. Adding the `Synthetic` flag here keeps both data sources clean:

- **BigQuery** owns release-level metadata (what *kind* of release this is)
- **YAML config** owns job-level mappings (which jobs belong to each release)

The `BuildSyntheticReleaseJobOverrides` function bridges them: it takes the
synthetic release names from BigQuery and the job mappings from YAML to produce
the overrides map.

### Phase 3: Fix `setRelease` (Variant Extraction)

**File:** `pkg/variantregistry/ocp.go`

Changes:
1. Add `syntheticReleaseJobOverrides map[string]string` to `OCPVariantLoader` struct.
2. Accept overrides as a parameter in `NewOCPVariantLoader`.
3. Add a synthetic override check at the **end** of `setRelease`, after all existing
   logic. This preserves the original matching behavior for non-synthetic jobs
   (config loop -> extractReleases overwrites) and simply overrides the final
   result for jobs in the synthetic overrides map:

```go
func (v *OCPVariantLoader) setRelease(logger logrus.FieldLogger, variants map[string]string, jobName string) {
    if presubmitRegex.MatchString(jobName) {
        variants[VariantRelease] = "Presubmits"
        return
    }

    // Existing logic: config lookup with name containment check (unchanged)
    for releaseName, releaseConfig := range v.config.Releases {
        if _, ok := releaseConfig.Jobs[jobName]; ok && strings.Contains(jobName, releaseName) {
            variants[VariantRelease] = releaseName
        }
    }

    // Existing logic: extract from job name (unchanged -- overwrites config loop)
    releasesInJobName := extractReleases(jobName)
    if count := len(releasesInJobName); count > 0 {
        release := releasesInJobName[count-1]
        variants[VariantRelease] = release.Original()
        variants[VariantReleaseMajor] = strconv.Itoa(release.Segments()[0])
        variants[VariantReleaseMinor] = strconv.Itoa(release.Segments()[1])
        // ... fromRelease and upgrade logic (unchanged)
    }

    // ... existing upgrade logic (unchanged)

    // Synthetic release overrides take priority over all other release logic.
    // Applied last so that Major/Minor/FromRelease/Upgrade fields are still
    // extracted from the job name by the logic above.
    if release, ok := v.syntheticReleaseJobOverrides[jobName]; ok {
        variants[VariantRelease] = release
    }
}
```

### Phase 4: Fix `VariantSnapshot.Identify`

**File:** `pkg/variantregistry/snapshot.go`

Changes:
1. Accept `syntheticReleaseJobOverrides` as a parameter in `NewVariantSnapshot`.
2. Track processed jobs to avoid double-processing:

```go
func (s *VariantSnapshot) Identify() (JobVariants, error) {
    newVariants := map[string]map[string]string{}
    variantSyncer := OCPVariantLoader{
        config: s.config, views: s.views,
        syntheticReleaseJobOverrides: s.syntheticReleaseJobOverrides,
    }

    for _, releaseCfg := range s.config.Releases {
        for job := range releaseCfg.Jobs {
            if isIgnoredJob(job) {
                continue
            }
            // Skip if already processed (job listed in multiple releases)
            if _, done := newVariants[job]; done {
                continue
            }
            newVariants[job] = variantSyncer.calculateVariantsForJob(
                s.log, job, nil, syntheticClusterDataOS(job))
        }
    }
    return newVariants, nil
}
```

### Phase 5: Wire Up Call Sites

Each command that needs synthetic overrides follows the same pattern:

1. Load releases from BigQuery via `api.GetReleasesFromBigQuery`
2. Extract synthetic release names from the `Synthetic` field
3. Call `v1.BuildSyntheticReleaseJobOverrides(config.Releases, syntheticReleaseNames)`
4. Pass the resulting map to the consumer

**Files:**
- `cmd/sippy/load.go` — `prowLoader()` builds overrides, passes to `prowloader.New`
- `cmd/sippy/variants_generate.go` — Builds overrides, passes to `NewOCPVariantLoader`
- `cmd/sippy/variants_snapshot.go` — Builds overrides, passes to `NewVariantSnapshot`
  (requires new `BigQueryFlags` and `GoogleCloudFlags`)

### Phase 6: Version-Dependent Variant Fallback

**File:** `pkg/variantregistry/ocp.go`

When `VariantRelease` is set to a synthetic release name (e.g., `rosa-stage`),
functions like `setContainerRuntime` and `setNetwork` that parse `VariantRelease`
as a version number will fail. Add a `releaseVersionFromVariants` helper that
falls back to `ReleaseMajor`/`ReleaseMinor` when `VariantRelease` is unparseable:

```go
func releaseVersionFromVariants(jLog logrus.FieldLogger, variants map[string]string) *version.Version {
    release, exists := variants[VariantFromRelease]
    if !exists {
        release, exists = variants[VariantRelease]
    }
    if exists {
        if v, err := version.NewVersion(release); err == nil {
            return v
        }
    }
    if major, ok := variants[VariantReleaseMajor]; ok {
        if minor, ok := variants[VariantReleaseMinor]; ok {
            if v, err := version.NewVersion(major + "." + minor); err == nil {
                return v
            }
        }
    }
    jLog.Warning("release version not found, unable to determine version-dependent variant")
    return nil
}
```

### Regexp Matching (No Change)

Synthetic releases also have `Regexp` patterns (e.g.,
`^periodic-ci-openshift-online-rosa-e2e-main-.*`). These regexps do **not** get
priority -- only explicit `Jobs` map entries are treated as synthetic claims.
Regexp matches continue to follow the existing iteration order in both
`processProwJob` and `setRelease`. If a regexp-matched job needs priority, it
should be added to the `Jobs` map explicitly.

## Behavioral Summary

### Before (Current)

```text
Job: periodic-ci-openshift-osde2e-main-nightly-4.22-osd-aws

processProwJob:
  Iterates releases -> matches 4.22 first (or rosa-stage, nondeterministic)
  Imported under: 4.22 (usually)

setRelease:
  Config check: rosa-stage.Jobs has the job, but "rosa-stage" not in job name -> skip
  Name extraction: finds "4.22" -> VariantRelease = "4.22"

Result: Job appears under 4.22 dashboard, NOT rosa-stage.
```

### After (Proposed)

```text
Job: periodic-ci-openshift-osde2e-main-nightly-4.22-osd-aws

processProwJob:
  syntheticReleaseJobOverrides["...-4.22-osd-aws"] = "rosa-stage" -> match
  Imported under: rosa-stage

setRelease:
  Config check + extractReleases run as before (setting Major/Minor/Upgrade)
  syntheticReleaseJobOverrides["...-4.22-osd-aws"] = "rosa-stage" -> override at end
  VariantRelease = "rosa-stage"
  VariantReleaseMajor = "4", VariantReleaseMinor = "22" (still extracted)

Result: Job appears under rosa-stage dashboard. Major/minor version info preserved.
```

## Migration and Backward Compatibility

1. **Zero impact on standard releases.** The `Synthetic` column defaults to
   `FALSE`. All existing OCP releases continue to work exactly as before.

2. **Config generator compatibility.** The external `sippy-config-generator`
   tool can continue adding jobs to OCP releases. The synthetic flag in BigQuery
   ensures the synthetic release wins. No changes needed in `openshift/ci-tools`.

3. **No re-import needed.** Once deployed, new data loads will route jobs to
   the correct release. Existing job runs imported under the wrong release
   will remain there until they age out naturally over the standard 2-week
   window.

4. **Variant snapshot regeneration.** Run `make update-variants` after
   deploying. The snapshot diff will show synthetic release jobs changing
   from OCP version releases (e.g., `Release: "4.22"`) to their synthetic
   release (e.g., `Release: "rosa-stage"`).

5. **BigQuery schema change.** The `ALTER TABLE` to add the `Synthetic` column
   must be run before deploying the code change. The column defaults to `FALSE`,
   so the code is safe to deploy even if the column exists but hasn't been
   populated yet — it will simply produce no overrides until the `UPDATE` runs.

## Resolved Questions

1. **Synthetic release regexps do NOT get priority.** Only explicit `Jobs` map
   entries in synthetic releases are treated as overrides. Regexps follow normal
   iteration order. If a regexp-matched job needs priority, add it to `Jobs`.

2. **TRT-2619 (Associated Owners with Releases) is out of scope.** The `Owner`
   variant already exists on the variant table and serves a similar purpose.
   No need to coordinate here.

3. **No warning for config generator overlap.** The config generator will
   continue adding rosa-stage jobs to OCP releases like `4.22`. This is
   expected and happens for every synthetic release job that has a version in
   its name. A warning would just be noise.

4. **Presubmits is not a synthetic release.** It uses regexp-based matching
   with no explicit `Jobs` entries and is handled by dedicated presubmit
   detection logic in `setRelease`.

5. **Why BigQuery, not YAML?** The YAML config is auto-generated and cannot be
   manually modified. BigQuery already stores release metadata and is the
   appropriate place for this flag.

6. **Overrides passed as parameters, not mutated onto config.** The existing
   codebase keeps BigQuery-sourced data (`[]sippyv1.Release`) and YAML-sourced
   data (`*SippyConfig`) as separate arguments passed to consumers. The
   overrides map follows this same pattern rather than mutating `SippyConfig`.
